# 炎上チェックスキル

技術記事が炎上するリスクを事前にチェックし、問題となりうる表現や内容を指摘するスキルです。

## 使い方

```
/check-controversy
```

対象の記事ファイルパスを指定するか、直近で編集中の記事を自動検出します。

## チェック手順

1. 対象記事ファイルを読み込む
2. 5つの炎上パターンに照らしてチェック
3. リスク評価と修正提案をレポート出力

---

## 炎上パターン1: 顧客不在のHard Things自慢

### 背景・事例

**Hacobu社の事例**: 「UIの色を変えただけで大量のクレームを頂戴した話」
- 社内の苦労話が中心で顧客への対応が薄かった
- 「困難を乗り越えた」美談だが、顧客は放置されていた印象
- 「結局クレームにどう対応したの？」という疑問が噴出

### チェック項目

- [ ] 失敗談・苦労話は「顧客にどう対応したか」が含まれているか
- [ ] 社内アピールが顧客視点より優先されていないか
- [ ] 「結局、顧客の問題は解決したのか」が明確か
- [ ] 被害者・影響を受けた人への配慮があるか

### 危険な表現パターン

```
NG: 「チームで協力して乗り越えました」（顧客への言及なし）
NG: 「大変でしたが良い経験になりました」（問題解決の言及なし）
NG: 「クレームが来ましたが、社内では〜」（顧客対応の説明なし）

OK: 「お客様には〜で対応し、〜という形で解決しました」
OK: 「この経験から、〜という改善を行いました」
```

## 炎上パターン2: マサカリ誘発表現

### 背景・事例

Qiita/Zennでは「マサカリ」と呼ばれる厳しい批判が飛んでくることがあります。
特に言語・フレームワークの比較記事や、断定的な主張は炎上しやすい傾向があります。

### チェック項目

- [ ] 「すべてのエンジニアは〜すべき」等の広い主語がないか
- [ ] 「〜が常識」「〜は間違い」等の断定表現がないか
- [ ] 言語・フレームワークの優劣比較をしていないか
- [ ] 私見は「私の経験では」「個人的には」と限定しているか
- [ ] 「〜は使うな」等の否定的な断言がないか

### 危険な表現パターン

```
NG: 「すべてのエンジニアはTypeScriptを使うべきだ」
NG: 「JavaScriptはもう時代遅れだ」
NG: 「Reactより Vue の方が優れている」
NG: 「この書き方を知らないのはありえない」
NG: 「〜は常識だ」「〜は当然だ」

OK: 「私のチームではTypeScriptを採用して良かったと感じています」
OK: 「この状況ではReactが適していると考えます」
OK: 「個人的には〜という印象を持っています」
```

### 特に注意が必要なトピック

- プログラミング言語の比較
- フレームワーク・ライブラリの比較
- エディタ・IDE の比較（Vim vs Emacs等）
- 開発手法の比較（アジャイル vs ウォーターフォール等）
- キャリア論・働き方論

## 炎上パターン3: 機密情報・個人情報漏洩

### チェック項目

- [ ] 社内プロジェクト名・システム名が伏せられているか
- [ ] 顧客名・取引先名が匿名化されているか
- [ ] スクリーンショットに個人情報が映っていないか
- [ ] 内部APIキー・認証情報がコードに含まれていないか
- [ ] 社内Slack・メールの内容が無断引用されていないか
- [ ] 社内の人物が特定できる記述がないか

### 危険な表現パターン

```
NG: 「A社との案件で〜」（取引先名の露出）
NG: 「弊社の〇〇システムでは〜」（内部システム名の露出）
NG: コード内に API_KEY="sk-xxx..." が含まれる
NG: スクリーンショットにメールアドレス・氏名が映っている

OK: 「とある顧客案件で〜」
OK: 「社内の基幹システムでは〜」
OK: コード内は API_KEY="your-api-key-here" とプレースホルダー
```

## 炎上パターン4: 上から目線・マウンティング

### チェック項目

- [ ] 「初心者はこれを知らない」等の見下した表現がないか
- [ ] 自社・自分の優位性を過度にアピールしていないか
- [ ] 他社・他者の批判になっていないか
- [ ] 「〜もできないの？」等の煽り表現がないか
- [ ] 読者を試すような表現がないか

### 危険な表現パターン

```
NG: 「これを知らないエンジニアはヤバい」
NG: 「こんな基本的なことも知らないの？」
NG: 「弊社はこんなにすごいことをやっている」
NG: 「〇〇社のやり方は間違っている」
NG: 「素人はこういうミスをしがち」

OK: 「私も最初は知りませんでしたが〜」
OK: 「意外と見落としがちなポイントとして〜」
OK: 「弊社で試してみた結果〜」
```

## 炎上パターン5: 誤情報・未検証情報

### チェック項目

- [ ] 動作確認せずに書いたコードがないか
- [ ] 古い情報を最新と誤認させていないか
- [ ] 出典のない統計データ・数値がないか
- [ ] 伝聞情報を事実として書いていないか
- [ ] セキュリティ上問題のあるコードがないか

### 危険な表現パターン

```
NG: 「〜らしいです」（出典なしの伝聞）
NG: 「〜だそうです」（未確認情報）
NG: 「〇〇％のエンジニアが〜」（出典なしの統計）
NG: 古いバージョンのコードを最新として紹介

OK: 「〇〇によると〜（リンク）」
OK: 「2024年1月時点の情報です」
OK: 「バージョン X.Y.Z で動作確認しています」
```

## リスク評価基準

| レベル | 説明 | 対応 |
|--------|------|------|
| 🔴 高リスク | 炎上する可能性が高い | **公開前に必ず修正** |
| 🟡 中リスク | 批判を受ける可能性あり | 修正を推奨 |
| 🟢 低リスク | 軽微な指摘の可能性 | 可能なら修正 |

---

## チェック出力フォーマット

```markdown
# 炎上チェック結果

## 総合リスク評価
🟡 中リスク

## 検出された問題

### 🔴 高リスク
1. **マサカリ誘発表現**（行XX）
   - 問題: 「すべてのエンジニアは〜すべき」
   - 修正案: 「私の経験では〜が効果的でした」

### 🟡 中リスク
1. **上から目線表現**（行XX）
   - 問題: 「〜を知らないのはありえない」
   - 修正案: 「意外と見落としがちな〜」

### 🟢 低リスク
- なし

## パターン別チェック結果

| パターン | 結果 | 詳細 |
|----------|------|------|
| 顧客不在のHard Things | ✅ OK | - |
| マサカリ誘発表現 | ⚠️ 要確認 | 2件 |
| 機密情報漏洩 | ✅ OK | - |
| 上から目線 | ⚠️ 要確認 | 1件 |
| 誤情報・未検証 | ✅ OK | - |
```
