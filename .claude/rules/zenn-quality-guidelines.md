# Zenn記事品質ガイドライン

Zennに公開する記事として適切なクオリティを満たすためのガイドラインです。

---

## Zennらしい記事の特徴

### 1. 読者フレンドリー

**冒頭で明示すべき内容：**
- 誰向けの記事か（対象読者）
- 読むと何が得られるか（メリット）
- 前提知識・環境

```markdown
<!-- 良い例 -->
この記事は、Claude Codeを導入したばかりのチームリーダー向けです。
チームへの定着を促進するための具体的な施策と、3ヶ月間の導入効果を共有します。

<!-- 悪い例 -->
Claude Codeについて書きます。
```

**TL;DR（まとめ）の配置：**
- 長い記事では冒頭または末尾にTL;DRを配置
- 箇条書きで要点を3-5個程度にまとめる

```markdown
## TL;DR

- Claude Codeの導入で開発速度が30%向上した
- 定着には「小さな成功体験」の共有が効果的
- 週1回の活用事例共有会を推奨
```

---

### 2. 謙虚なトーン

Zennコミュニティでは、謙虚で建設的なトーンが好まれます。

**推奨する表現：**

| 避けるべき表現 | 推奨する表現 |
|----------------|--------------|
| 〜すべきです | 〜するのがおすすめです |
| 〜は間違いです | 〜は別の方法もあります |
| 絶対に〜 | 多くの場合〜 |
| 〜が正解です | 〜が効果的でした |
| 〜を使わないのはおかしい | 〜を試してみる価値があります |

**良いトーンの例：**

```markdown
<!-- 良い例 -->
私たちのチームではこの方法が効果的でしたが、
チームの状況によって最適な方法は異なるかもしれません。

<!-- 悪い例 -->
このやり方以外は全て間違いです。
今すぐ導入すべきです。
```

**自分の経験として語る：**
- 「〜と言われています」より「〜を試しました」
- 「〜が常識です」より「〜が効果的でした」
- 主語を明確に（「私は」「我々のチームでは」）

### 3. Zenn独自記法の活用

Zenn独自の記法を効果的に使うことで、読みやすさが向上します。

**:::message の使い所：**
- 重要なポイントの強調
- 注意事項・前提条件の明示
- 補足情報

```markdown
:::message
この方法はNode.js v18以上が必要です。
:::

:::message alert
本番環境で実行する前に、必ずバックアップを取ってください。
:::
```

**:::details の使い所：**
- 長いコードブロック
- 詳細な設定ファイル
- 補足的な説明（本筋から外れる内容）
- エラーログの全文

```markdown
:::details 完全な設定ファイル
```json
{
  "compilerOptions": {
    // 長い設定内容...
  }
}
```
:::
```

**コードブロックの工夫：**
- ファイル名を付ける（`typescript:src/index.ts`）
- diffハイライトでBefore/Afterを示す
- 実行可能な完全なコードを載せる

```markdown
```diff typescript:src/config.ts
- const timeout = 5000;
+ const timeout = 10000; // タイムアウトを延長
```
```

### 4. コミュニティへの貢献意識

Zennは技術知見を共有するコミュニティです。以下の姿勢で執筆しましょう。

**共有する動機を明確に：**
- 「自分がハマったから共有」
- 「ドキュメントになかったから書いた」
- 「チームで効果があったから紹介」

**再現可能性を重視：**
- バージョン情報を明記
- 環境・前提条件を書く
- コピペで動くコードを目指す

**フィードバックを歓迎する姿勢：**
- 「より良い方法があればコメントで教えてください」
- 「間違いがあればご指摘ください」

### 5. AI文章っぽさを避ける

AIが書いた文章は特有のパターンがあり、読者に見抜かれやすい。
人間らしい文章にするための観点。

**避けるべきAI文章の特徴：**

| パターン | 例 | 改善案 |
|----------|-----|--------|
| 過剰な箇条書き | 何でもリスト化 | 文章で流れを作る |
| 定型フレーズ | 「〜について解説します」 | 自然な導入に |
| 曖昧な修飾語 | 「様々な」「非常に」 | 具体的な表現に |
| 両論併記の癖 | 必ず「一方で〜」 | 自分の意見を明確に |
| 丸い数字 | 「約30%向上」 | 実測値を使う |

**人間らしさを出すポイント：**

- 具体的なエピソード・失敗談を入れる
- 固有名詞を使う（ツール名、具体的な状況）
- 感情・本音を入れる（「正直面倒だった」「これは最高」）
- 試行錯誤の過程を書く（「最初は〇〇を試したけど〜」）
- 自分の好み・意見を明確にする

## レビュー時のチェックポイント

記事レビュー時に以下を確認してください。

### 必須チェック

- [ ] 冒頭で対象読者と得られるものが明示されている
- [ ] 断定的・攻撃的な表現がない
- [ ] 技術的な主張に根拠（経験・データ）がある
- [ ] 個人・所属会社を特定できる情報がない

### 推奨チェック

- [ ] TL;DRまたはまとめセクションがある
- [ ] :::message や :::details を効果的に使っている
- [ ] コードブロックにファイル名が付いている
- [ ] 「〜してみた」「〜が効果的でした」など体験ベースの表現
- [ ] バージョン情報・環境情報が明記されている

### トーンチェック

以下の表現がないか確認：

```
🔴 避けるべき表現
- 「〜すべき」「〜しなければならない」
- 「〜は間違い」「〜はダメ」
- 「常識として」「当然」
- 「〜を知らないのはおかしい」
- 「〇〇（技術）は終わった」「〇〇はオワコン」
```

### AI文章チェック

以下のAI文章パターンがないか確認：

- [ ] 過剰な箇条書き・リスト化をしていない
- [ ] 「〜について解説します」等の定型フレーズがない
- [ ] 「様々な」「非常に」「多くの」を多用していない
- [ ] 具体的なエピソード・数値がある
- [ ] 自分の感情・意見が含まれている

## Zenn記事の良い例

### 構成例

```markdown
---
title: "Claude Codeをチームに導入して3ヶ月で見えた効果と課題"
emoji: "🤖"
type: "tech"
topics: ["claude", "ai", "productivity", "devops"]
published: true
---

## TL;DR

- 導入3ヶ月で開発速度が30%向上
- 定着には週次の事例共有が効果的
- 課題はコードレビューとの併用

## はじめに

この記事は、AIコーディングツールの導入を検討しているチームリーダー向けです。
私たちのチーム（5人、Webアプリ開発）でClaude Codeを導入した経験を共有します。

:::message
この記事の内容は2024年12月時点のものです。
Claude Codeのバージョン: 1.0.x
:::

## 導入の背景

（以下、本文...）

## おわりに

私たちのチームでは効果がありましたが、チームの規模や開発スタイルによって
最適な導入方法は異なると思います。

より良い方法や、導入時の工夫があればぜひコメントで教えてください。
```
